Setting Up Your Tools

Before we start — what do you need?

You only need two things: a web browser and a proxy tool. That's it.

1️⃣ Install Firefox + FoxyProxy
  1. Download Firefox from firefox.com (free)
  2. Install FoxyProxy extension from Firefox Add-ons
  3. FoxyProxy lets you switch traffic to Burp Suite
2️⃣ Install Burp Suite Community
  1. Go to portswigger.net/burp/communitydownload
  2. Download Burp Suite Community Edition (free)
  3. Open Burp → "Temporary project" → "Use Burp defaults"
  4. Go to Proxy tab → Options → note port (usually 8080)
What is Burp Suite? It's a middleman between your browser and websites. It intercepts traffic so you can see, modify, and replay requests. Think of it as a "pause button" for web traffic.
3️⃣ Connect Firefox to Burp
  1. Open FoxyProxy → Add new proxy → Host: 127.0.0.1, Port: 8080
  2. Enable the proxy in FoxyProxy
  3. In Firefox, visit http://burp → click "CA Certificate"
  4. Import certificate into Firefox: Settings → Privacy → Certificates → Import
4️⃣ Test It Works
  1. Turn on FoxyProxy in Firefox
  2. Visit any website
  3. Go to Burp → Proxy → HTTP History
  4. You should see the request appear! 🎉
💡 Remember: Always turn OFF the proxy when not testing. It slows down normal browsing.

SQL Injection (SQLi)
Advanced

🧠 What is SQL Injection?

When you type into a search box or login form, the website sends your input to a database using SQL. SQL Injection happens when the website doesn't check your input — so you can trick the database into doing things it shouldn't.

Imagine you tell a waiter your name is "; DROP TABLE customers;--. The waiter writes it on a form, and the kitchen deletes the entire customer database. That's SQL Injection — your "name" became a command.
💥 What happens? An attacker can read all data (passwords, credit cards, messages), delete data, or take control of the server.
🎯 Where to find it
  • Login forms — try ' in username field
  • Search bars — try test' OR '1'='1
  • URL parameters — if URL has ?id=1, try ?id=1' OR '1'='1
  • Anything with a number — product IDs, page numbers
🔬 How to test
  1. Find an input field — login, search, URL with ?id=
  2. Type a single quote ' and press Enter. If you see a database error, it might be vulnerable!
  3. Try: ' OR '1'='1 — if you get logged in without password, it's vulnerable.
  4. Try in URL: ?id=1' OR '1'='1 — if you see extra results, it's vulnerable.

Cross-Site Scripting (XSS)
Intermediate

🧠 What is XSS?

XSS means you can inject JavaScript code into a website that other people see. When a victim visits the page, their browser runs YOUR code. This lets you steal cookies (login tokens) or redirect them.

A bulletin board where anyone posts notes. If the board doesn't check what people write, someone could post: "Everyone who reads this: send your house keys to this address." That's XSS — malicious code hidden in a trusted place.
💥 What happens? Attacker steals login sessions, redirects users to phishing sites, changes page content, or takes over accounts.
🎯 Where to find it
  • Search bars — does your search term appear on results page?
  • Comments/forums — can you post code in a comment?
  • Profile fields — bio, name, website URL
  • Error pages — visit fake URL, does URL appear in error?
  • URL parameters — does ?name=test show "test" on page?
🔬 How to test
  1. Find an input — search bar, comment box, URL param
  2. Type a unique marker like XYZTEST123 and submit
  3. Look at the page — does XYZTEST123 appear? Input is reflected!
  4. Now try: <script>alert(1)</script>
  5. If popup appears with number 1 → XSS confirmed! 🎉
  6. If blocked — try: <img src=x onerror=alert(1)>
💉 Beginner Payloads

IDOR (Accessing Other People's Data)
Beginner Friendly

🧠 What is IDOR?

The website uses a number (like an ID) to identify your data, but doesn't check if you OWN that data. If your profile is /profile?id=100, try /profile?id=101 — you might see someone else's profile!

Hotel room keys with just the room number. If your key opens room 100, try 101, 102... you can walk into anyone's room. The hotel should check YOUR key matches YOUR room.
💥 What happens? Attacker views other users' private data — profiles, messages, invoices, payment info.
🎯 Where to find it
  • Profile pages — /profile?id=123 → change number
  • API calls — look for numbers in URLs or JSON
  • File downloads — PDF/image links with IDs
  • Orders/invoices — /order/INV-001 → try INV-002
  • Mobile apps — intercept API calls, look for user IDs
🔬 How to test
  1. Create two accounts — "attacker" and "victim"
  2. Login as victim — do something that creates data (upload, message, order)
  3. Note the URL/API call in Burp — look for the ID number
  4. Login as attacker — try accessing same URL with victim's ID
  5. If you see victim's data → IDOR confirmed! 🎉
💉 What to try
📋 Simple tests 
-- If your data is at:
/profile?id=100

-- Try:
/profile?id=101
/profile?id=99
/profile?id=1
/profile?id=0

-- Also try:
/api/user/100  →  /api/user/101
/download?file=report_100.pdf  →  report_101.pdf
💡 Beginner Tip: IDOR is the #1 easiest vulnerability to find with high payouts ($500-$5000+). Always test every number in a URL!

Open Redirect
Beginner Friendly

🧠 What is Open Redirect?

Some websites redirect you after login/logout using a "return URL" parameter. If the website doesn't check WHERE it redirects, an attacker can send users to a phishing site.

A security guard asks "which floor?" and takes you there without checking. An attacker says "floor 100" (doesn't exist) — the guard just goes there anyway.
💥 What happens? Used in phishing — victim clicks trusted link but gets sent to attacker's fake login page. Also steals OAuth tokens.
🎯 Where to find it
  • Logout links — /logout?redirect=/home
  • Login redirect — /login?next=/dashboard
  • Common params: redirect, url, next, return, dest, goto
🔬 How to test
  1. Find a redirect parameter — look for ?redirect=, ?next=, ?url=
  2. Change value to https://evil.com
  3. Visit URL — if redirected to evil.com, it's open redirect!

💡 Beginner Tip: Open redirect is simple to find. Look for it on every website — most have logout redirects or OAuth callbacks.

Information Disclosure
Beginner Friendly

🧠 What is Info Disclosure?

Websites sometimes accidentally expose sensitive files, debug info, or config details. These can reveal passwords, API keys, and internal paths. You don't need to "hack" — the data is just sitting there in the open.

A company throws away filing cabinets but forgets to remove the files. Anyone can open them and read everything.
💥 What happens? Attackers find API keys, database passwords, admin credentials and use them to break in.
🎯 Files to check (just visit these URLs!)
📋 No tools needed — just paste in browser 
https://target.com/robots.txt
https://target.com/.git/config
https://target.com/.env
https://target.com/.DS_Store
https://target.com/web.config
https://target.com/sitemap.xml
https://target.com/server-status
https://target.com/phpinfo.php
https://target.com/backup.zip
https://target.com/.htaccess
https://target.com/debug
https://target.com/api/
https://target.com/graphql
💡 Beginner Tip: EASIEST vulnerability to find. Type these URLs into your browser. If you see anything interesting, screenshot and report!
🔬 What to look for
  • robots.txt — "Disallow: /admin" reveals hidden pages
  • .git/config — source code might be downloadable!
  • .env — database passwords, API keys, secrets
  • phpinfo.php — PHP version, server info
  • Error pages — trigger error, look for stack traces

CSRF (Tricking Your Browser)
Intermediate

🧠 What is CSRF?

CSRF tricks YOUR browser into doing something on a website you're logged into — without your knowledge. If you're logged into your bank and visit a malicious page, it could transfer money from YOUR account.

Your friend gives you a signed check (your browser has cookies). A stranger picks it up, writes "pay me $1000" and cashes it — because it has your valid signature.
💥 What happens? Attacker changes your email, password, transfers money, posts messages, or deletes your account.
🔬 How to test
  1. Login to target and go to sensitive action (change password/email)
  2. Look at request in Burp — does it have a CSRF token?
  3. If NO token → CSRF confirmed! Make a PoC HTML page
  4. If there IS a token — try removing it. Does it still work?
💉 PoC HTML
📋 Save as .html and open while logged into target 
<html><body>
  <form method="POST" action="https://target.com/change-email">
    <input type="hidden" name="email" value="attacker@evil.com">
    <input type="submit" value="Click me">
  </form>
  <script>document.forms[0].submit();</script>
</body></html>

SSRF (Server-Side Trick)
Advanced

🧠 What is SSRF?

SSRF tricks the SERVER into making requests to places it shouldn't — like internal services or cloud metadata APIs. The server trusts itself, so it can access things you can't.

You ask a bank employee to "fetch a document from this URL." You gave them a URL to the bank's internal vault camera. Since they're an employee, they have access — and bring you the footage.
💥 What happens? Can steal cloud credentials (AWS/GCP keys), access internal databases, scan networks.
🎯 Where to find it
  • "Import from URL" features
  • Image loading by URL — profile pictures
  • PDF generators — HTML to PDF
  • Webhook URLs
  • File upload by URL
🔬 How to test
  1. Set up listener — go to webhook.site (free, gives you a URL)
  2. Find a URL input — any field that takes a URL
  3. Paste your webhook URL and submit
  4. Check webhook.site — if request appears, SSRF confirmed!
  5. Try internal URLs: http://127.0.0.1, http://localhost

Authentication Bypass
Intermediate

🧠 What is Auth Bypass?

You can access accounts or admin panels without knowing the password. Includes trying default passwords, resetting passwords insecurely, or skipping 2FA.

A bouncer checks your ID. But if you show a fake "VIP" ID, he just lets you in. The security check exists but doesn't actually work.
💥 What happens? Full account takeover — attacker accesses any account, including admin with full control.
🔬 Things to try
📋 Simple checks 
-- 1. Default credentials
admin:admin
admin:password
admin:123456
root:root
test:test

-- 2. SQLi in login
Username: admin'--
Password: anything

-- 3. Admin panels
/admin  /admin/login  /wp-admin  /administrator  /console

-- 4. 2FA bypass
# Login → reach 2FA page
# Try going directly to /dashboard
# Or remove 2FA code from request
💡 Beginner Tip: Always try default credentials! You'd be surprised how many sites use admin:admin.

Local File Inclusion (LFI)
Intermediate

🧠 What is LFI?

LFI lets you read files from the server by manipulating file paths. If a website loads pages based on a URL parameter (like ?page=about), you can trick it into loading system files like /etc/passwd.

A librarian fetches books by title. You ask for "the secret employee handbook" and they bring it — they never check if you're allowed to read it.
💥 What happens? Can read passwords, API keys, config files, source code, private keys. Can lead to full server compromise.
🎯 Where to find it
  • Page params: ?page=home, ?file=about, ?template=contact
  • File downloads: ?doc=report.pdf
  • Language selectors: ?lang=en
🔬 How to test
  1. Find a page parameter — ?page=home
  2. Try traversal: ?page=../../../../etc/passwd
  3. If you see file contents (like "root:x:0:0:...") → LFI confirmed!
  4. If blocked: try double encoding: ..%252f..%252f..%252fetc/passwd
