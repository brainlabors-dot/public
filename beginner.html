👋 Welcome — Start Here!

New to bug bounty? No problem. Here's everything you need to know before diving in:

  • What is Bug Bounty? — Companies pay you to find security bugs in their websites and apps. You report the bug, they pay you.
  • Do I need to be a programmer? — Not necessarily! Many bugs can be found with basic browser skills and curiosity.
  • How much can I earn? — $50 for simple bugs, $500–$5,000+ for serious ones. Some critical bugs pay $10,000–$50,000+.
  • What's the first step? — Pick ONE beginner vulnerability from below, learn it, and practice on your own test apps or public bug bounty programs.
  • Is it legal? — YES, but ONLY on programs that explicitly allow it. Never test without permission. Read the program's scope and rules.
  • What tools do I need? — A browser, Burp Suite Community (free), and curiosity. That's it to start!

🔑 IDOR — Insecure Direct Object Reference

Access other users' data by changing a number in the URL
★☆☆ 💰 ▼
📖 What Is It?

IDOR happens when an application uses a predictable number (like an ID) in the URL to access data, and doesn't check if you're allowed to see that data. Think of it like apartment numbers — if I know your apartment is #404, I can just walk there. The building has no lock checking if I belong.

🔍 When to Look
  • URLs that contain numbers like /user/123/profile, /order/456, /invoice/789
  • Download links with IDs: /download?id=1234
  • Any feature where you access your own data (profile, orders, documents)
  • APIs with numeric IDs in GET/PUT/DELETE requests
🧪 How to Test
  • Create two accounts (Account A and Account B)
  • Log in as Account A and go to your profile page
  • Look at the URL — you'll see something like /user/1001
  • Change the number: /user/1002 and press Enter
  • If you see Account B's data → You found an IDOR!
  • Try this with orders, documents, invoices, any user data
👁️ What to Look For
  • You can see another user's profile, orders, or private data
  • Changing the ID number shows different data without any error
  • No "Access Denied" or "Not Authorized" message
💥 Impact: Attackers can access ANY user's private data. This includes personal info, financial records, medical data, or documents. This is a privacy violation and often a compliance issue (GDPR, HIPAA).
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Include both URLs (the one that works and the one you shouldn't access), the steps to reproduce, and a screenshot showing another user's data. Mention the potential data exposure.
02

🔀 Open Redirect

Trick the website into redirecting users to a malicious site
★☆☆ 💰 ▼
📖 What Is It?

An open redirect means a website has a link that sends you somewhere else, but it doesn't verify WHERE. It's like a taxi driver who takes you anywhere you ask — even to a dark alley. Attackers use this for phishing: "Click this link to yourbank.com" → but it actually redirects to a fake site that steals your password.

🔍 When to Look
  • URLs with parameters like ?redirect=, ?url=, ?next=, ?return=, ?dest=
  • Login pages that redirect you after authentication
  • "Continue reading" or "Share" links
  • Password reset flows and OAuth flows
🧪 How to Test
  • Find a URL with a redirect parameter
  • Change the redirect value to an external site: ?redirect=https://evil.com
  • Check if you get redirected to that external site
  • Try variations: use //evil.com, https://evil.com, or encoded versions
  • If the site redirects to your external URL → Open Redirect!
👁️ What to Look For
  • The browser URL bar changes to an external domain
  • No warning message about leaving the site
  • The redirect happens automatically without user confirmation
💥 Impact: Used in phishing attacks to steal credentials. An attacker sends "Click to reset your password at real-bank.com" but it redirects to a fake login page. The link looks legitimate because it starts with the real domain.
🛠️ Tools Needed
🌐 Browser
📝 Report Tip: Show the full URL with the malicious redirect, demonstrate the redirect happening, and explain how it could be used for phishing. Include a screenshot of the redirect chain.
03

🛡️ Missing Security Headers

Website is missing protections that prevent common attacks
★☆☆ 💰 ▼
📖 What Is It?

Security headers are like seatbelts for websites. They're instructions the server sends to the browser saying "hey, protect against these attacks." If they're missing, the site is more vulnerable. This is usually low-severity but easy to find and report.

🔍 When to Look
  • Every website you visit — this is universal
  • Look for: X-Frame-Options, Content-Security-Policy, X-Content-Type-Options, Strict-Transport-Security, X-XSS-Protection
  • Any new target you start testing
🧪 How to Test
  • Open Browser DevTools (F12) → Network tab
  • Refresh the page and click the first request
  • Look at the Response Headers
  • Check if these headers are present: X-Frame-Options, Content-Security-Policy, X-Content-Type-Options: nosniff, Strict-Transport-Security
  • If any are missing → that's a finding
👁️ What to Look For
  • Response headers are missing important security directives
  • No Content-Security-Policy at all
  • No X-Frame-Options (site can be framed/clickjacked)
💥 Impact: Makes other attacks easier. Missing headers can enable clickjacking, XSS, MIME sniffing attacks, and man-in-the-middle attacks. Each missing header is a different risk.
🛠️ Tools Needed
🌐 Browser DevTools 🔍 securityheaders.com
📝 Report Tip: List each missing header, explain what protection it provides, and link to the OWASP documentation. Keep it organized and reference specific risks.
04

📋 Information Disclosure

Website reveals sensitive technical details
★☆☆ 💰 ▼
📖 What Is It?

The website tells you too much about itself — server software, version numbers, internal file paths, stack traces, or debug information. It's like someone telling a stranger their home address, alarm code, and daily schedule. This info helps attackers plan bigger attacks.

🔍 When to Look
  • Error pages (trigger 404, 500 errors)
  • Response headers (Server, X-Powered-By)
  • Source code of pages (HTML comments, JavaScript)
  • Debug or verbose error messages
🧪 How to Test
  • Check response headers for Server and X-Powered-By values
  • Visit a non-existent page to trigger a 404 error
  • Look for stack traces or detailed error messages
  • View page source (Ctrl+U) and look for HTML comments
  • Check /robots.txt, /.git/config, /debug, /server-status
👁️ What to Look For
  • Server version disclosed (Apache/2.4.41, nginx/1.18.0)
  • Stack traces showing internal file paths
  • HTML comments with developer notes or credentials
  • Debug mode enabled in production
💥 Impact: Helps attackers identify specific software versions with known vulnerabilities. A stack trace can reveal database structure. HTML comments might contain hardcoded credentials. This is reconnaissance for bigger attacks.
🛠️ Tools Needed
🌐 Browser DevTools 🔧 curl
📝 Report Tip: Screenshot the disclosure, redact any actual credentials, and explain what an attacker could do with this information.
05

🖱️ Clickjacking

Trick users into clicking something they didn't mean to
★☆☆ 💰 ▼
📖 What Is It?

Clickjacking is hiding a button or link on top of something innocent. You think you're clicking "Play Video" but you're actually clicking "Delete Account." The attacker puts an invisible iframe over a button. It's like putting a transparent "Buy" button over a "Free Sample" button.

🔍 When to Look
  • Any website that doesn't use X-Frame-Options or CSP frame-ancestors
  • Sites with sensitive actions (delete, change password, transfer funds)
  • Any site that can be loaded in an iframe
🧪 How to Test
  • Create a simple HTML file on your computer
  • Use an <iframe> tag to embed the target website
  • If the site loads inside the iframe → it might be vulnerable
  • Use opacity: 0 on the iframe to make it invisible
  • Put a fake button over where a real button would be
  • If it works → Clickjacking confirmed!
👁️ What to Look For
  • The target site loads inside an iframe
  • No X-Frame-Options header in the response
  • No CSP frame-ancestors directive
💥 Impact: Attackers can trick users into performing actions without consent: clicking "Delete," "Transfer Money," "Change Password," or "Follow" buttons. Social media platforms are common targets.
🛠️ Tools Needed
🌐 Browser 📝 Text Editor
📝 Report Tip: Provide the PoC HTML file, explain what action an attacker could trick a user into performing, and recommend adding X-Frame-Options or CSP frame-ancestors.
06

🌐 CORS Misconfiguration

Website trusts any website to make requests on its behalf
★☆☆ 💰 ▼
📖 What Is It?

CORS (Cross-Origin Resource Sharing) tells browsers which websites are allowed to make requests to a server. If a site says "everyone is allowed," an attacker's website can steal data from logged-in users. It's like a bank that lets anyone walk into any account because "they said they're family."

🔍 When to Look
  • APIs that return user-specific data
  • Any site that uses cookies for authentication
  • Look at the Origin and Access-Control-Allow-Origin headers
🧪 How to Test
  • Open DevTools → Network tab
  • Make a request to the API
  • Look at the request header Origin and response header Access-Control-Allow-Origin
  • If Access-Control-Allow-Origin: * → might be vulnerable
  • If it reflects your Origin header (e.g., Origin: evil.com → Access-Control-Allow-Origin: evil.com) → vulnerable!
  • Also check for Access-Control-Allow-Credentials: true
👁️ What to Look For
  • Origin header is reflected in Access-Control-Allow-Origin
  • Access-Control-Allow-Origin: *
  • Access-Control-Allow-Credentials: true combined with wildcard
💥 Impact: Attackers can steal sensitive data from authenticated users by having them visit a malicious page. This can leak personal information, tokens, and private data.
🛠️ Tools Needed
🌐 Browser DevTools 🔧 curl
📝 Report Tip: Include the curl command showing the headers, explain that the Origin is reflected, and describe what data could be stolen.
07

📧 Email Header Injection

Inject extra headers into emails sent by the application
★☆☆ 💰 ▼
📖 What Is It?

When a website sends emails (contact forms, password resets), it puts your input into email headers. If you can inject newlines, you can add your own email headers — like CC'ing yourself, changing the subject, or sending to additional recipients. It's like slipping extra names into a conference call invite.

🔍 When to Look
  • Contact forms
  • Password reset email functionality
  • Invitation/invite features
  • Any form that sends an email
🧪 How to Test
  • Find a form that sends emails (contact, feedback, invite)
  • In the email field, try: test@test.com%0d%0aCc: victim@email.com
  • Also try: test@test.com%0d%0aBcc: attacker@email.com
  • Try injecting a new Subject: %0d%0aSubject: Hacked
  • If the extra header is accepted → Email Header Injection!
👁️ What to Look For
  • Additional recipients receive the email
  • The email subject changes from what was expected
  • The email contains injected headers in the raw source
💥 Impact: Attackers can spam people using your mail server, send phishing emails that look like they come from you, or BCC themselves on sensitive communications.
🛠️ Tools Needed
🌐 Browser
📝 Report Tip: Show the injected email received by the additional recipient, the raw email headers, and explain how the injection was performed.
08

🔧 Verb Tampering

Bypass access controls by changing the HTTP method
★☆☆ 💰 ▼
📖 What Is It?

HTTP has different "methods" (GET, POST, PUT, DELETE, OPTIONS, etc.). Sometimes a website blocks access using GET but forgets to block other methods. It's like a door with a "No Entry" sign that only blocks the front door — but you can walk in through the back.

🔍 When to Look
  • When you get a 403 Forbidden or 401 Unauthorized
  • Any endpoint that blocks certain actions
  • Admin panels or restricted areas
🧪 How to Test
  • Try accessing a restricted page with GET → 403 Forbidden
  • Now try with POST → check if you get 200 OK
  • Try PUT, DELETE, PATCH, OPTIONS, HEAD
  • Use Burp Suite to change the method easily
  • If a different method returns 200 → Verb Tampering!
👁️ What to Look For
  • GET returns 403 but POST returns 200
  • You can access restricted pages using unexpected methods
💥 Impact: Bypasses access controls to reach admin panels, restricted pages, or perform unauthorized actions.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 curl
📝 Report Tip: Show both requests (the blocked one and the one that bypasses), the responses, and what restricted content can be accessed.
09

📂 Forced Browsing

Access pages that aren't linked anywhere on the site
★☆☆ 💰 ▼
📖 What Is It?

Forced browsing means guessing or finding URLs that aren't linked from the website — admin panels, backup files, configuration pages, debug endpoints. It's like finding an unlocked door in the back of a building that nobody told you about.

🔍 When to Look
  • Any new website you're testing
  • Admin panels (/admin, /administrator, /wp-admin)
  • Backup files (/backup.zip, /site.bak)
  • Config files (/.env, /config.php, /web.config)
  • Debug pages (/debug, /trace.axd)
🧪 How to Test
  • Try common paths: /admin, /login, /dashboard, /test, /debug
  • Try backup files: /backup, /backup.zip, /db.sql
  • Try config files: /.env, /config.json, /wp-config.php.bak
  • Check /robots.txt for hidden paths
  • Use a wordlist with tools like dirb, dirbuster, or ffuf
👁️ What to Look For
  • Pages return 200 OK instead of 404 or 403
  • Admin panels accessible without login
  • Backup or config files downloadable
💥 Impact: Can expose admin panels, configuration files with credentials, database backups, or debug information. The impact depends on what content is found.
🛠️ Tools Needed
🌐 Browser 🔧 ffuf (free) 🔧 dirb (free)
📝 Report Tip: List the URLs found, their HTTP status codes, and what sensitive information they expose.
10

📁 Directory Traversal (Basic)

Read files on the server by manipulating file paths
★☆☆ 💰 ▼
📖 What Is It?

Directory traversal lets you access files outside the website's folder by using ../ (go up one folder). It's like using "go back" in a file explorer — if the website doesn't check, you can go back far enough to read system files like /etc/passwd.

🔍 When to Look
  • URLs that load files: ?page=about, ?file=report.pdf
  • Download links: /download?file=document.pdf
  • Template parameters: ?template=home
  • Any parameter that seems to reference a file
🧪 How to Test
  • Find a parameter that loads a file
  • Try: ?file=../../../../etc/passwd
  • Also try: ?file=....//....//....//etc/passwd (bypass filters)
  • Try: ?file=..%2f..%2f..%2fetc%2fpasswd (URL encoded)
  • On Windows: ?file=..\..\..\windows\system32\drivers\etc\hosts
  • If you see file contents → Directory Traversal!
👁️ What to Look For
  • Contents of /etc/passwd displayed
  • Any system file contents shown
  • Application source code visible
💥 Impact: Attackers can read sensitive server files including configuration files, credentials, private keys, and user data. Can lead to full server compromise.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Show the URL with the traversal payload, the file contents retrieved, and explain what sensitive data was exposed. Be careful not to include actual credentials in your report.

⚡ Reflected XSS (Cross-Site Scripting)

Inject scripts that execute when another user clicks your link
★★☆ 💰💰 ▼
📖 What Is It?

XSS means you can inject JavaScript code into a website that runs in other users' browsers. "Reflected" means the code is in the URL — when someone clicks your crafted link, the script executes in their browser. It's like writing instructions on a piece of paper and someone else reads and follows them.

🔍 When to Look
  • Search boxes that show your query in the results page
  • Error messages that include your input
  • URL parameters that appear on the page
  • Any input that gets "reflected" back without proper encoding
🧪 How to Test
  • Type <script>alert(1)</script> in the search box
  • If an alert pops up → Basic XSS!
  • If blocked, try: <img src=x onerror=alert(1)>
  • Try: <svg/onload=alert(1)>
  • Check if your input appears in the page source unencoded
  • Test all input fields: search, forms, URL parameters
👁️ What to Look For
  • An alert popup appears
  • Your HTML/script code appears in the page source unencoded
  • The browser renders your HTML tags
💥 Impact: Steal session cookies, redirect users to malicious sites, deface pages, or perform actions as the victim. Social media platforms are high-value targets.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Include the full URL with the payload, a screenshot of the alert, and a video if possible. Explain the impact and suggest adding output encoding.
12

🔄 CSRF (Cross-Site Request Forgery)

Force users to perform actions without their knowledge
★★☆ 💰💰 ▼
📖 What Is It?

CSRF tricks a logged-in user's browser into making a request they didn't intend. If you're logged into your bank, visiting a malicious page could trigger a money transfer. The bank trusts your browser because you're logged in. It's like someone forging your signature on a check because they know you have a bank account.

🔍 When to Look
  • Actions that change data (change password, email, profile, transfer)
  • Forms that use POST requests
  • Any state-changing action that doesn't require re-authentication
  • Check if CSRF tokens are present in forms
🧪 How to Test
  • Find a form that changes settings (change email, password, profile)
  • Check if there's a CSRF token in the form
  • If no token → Try submitting the form from a different domain
  • Create a local HTML page with an auto-submitting form pointing to the target
  • If the action succeeds → CSRF confirmed!
👁️ What to Look For
  • No CSRF token in forms
  • Action succeeds when submitted cross-origin
  • No SameSite cookie attribute
💥 Impact: Change email/password, transfer funds, delete accounts, make purchases, or modify settings — all without the user knowing. High impact on financial and e-commerce sites.
🛠️ Tools Needed
🌐 Browser 📝 Text Editor 🔧 Burp Suite (free)
📝 Report Tip: Include the PoC HTML, explain the vulnerable action, show the action succeeding cross-origin, and recommend CSRF tokens + SameSite cookies.
13

🔗 SSRF (Server-Side Request Forgery) — Basic

Make the server fetch resources from internal network
★★☆ 💰💰 ▼
📖 What Is It?

SSRF tricks the server into making requests to places it shouldn't — like internal services, databases, or cloud metadata. You tell the server "fetch this URL" and it does, even if it's an internal resource. It's like asking a mailroom clerk to deliver a letter to the CEO's office — they have access you don't.

🔍 When to Look
  • URL inputs: "Enter a URL to fetch/preview/import"
  • Webhook configurations
  • Image/file URL fetchers
  • PDF generators that fetch remote content
  • Any feature that takes a URL and processes it
🧪 How to Test
  • Set up a simple listener (use Burp Collaborator or a webhook site)
  • Enter your listener URL as the target URL
  • If the server makes a request to your listener → basic SSRF!
  • Try internal addresses: http://127.0.0.1, http://localhost, http://169.254.169.254
  • Check if internal resources are fetched
👁️ What to Look For
  • Your listener receives a request from the server's IP
  • Internal addresses are fetched successfully
  • Error messages reveal internal network information
💥 Impact: Access internal services, databases, cloud metadata (AWS/GCP/Azure credentials), scan internal networks, and potentially achieve remote code execution. Can lead to full infrastructure compromise.
🛠️ Tools Needed
🔧 Burp Suite (free) 🌐 Webhook.site (free)
📝 Report Tip: Show the request that triggered SSRF, the response from your listener, and explain what internal resources could be accessed. Escalate the impact if cloud metadata is reachable.
14

💉 SQL Injection — Basic

Inject database commands through user input
★★☆ 💰💰 ▼
📖 What Is It?

SQL injection happens when user input is inserted directly into a database query. By adding special characters, you can change what the query does — like making a login form bypass authentication or dumping all user data. It's like telling the librarian "find books by Author; also show me ALL books including the restricted ones."

🔍 When to Look
  • Login forms
  • Search functionality
  • URL parameters with IDs: ?id=1
  • Any input that goes to a database
  • Error messages showing database errors
🧪 How to Test
  • In a search box, type: ' OR '1'='1
  • In a login form, try: admin' -- as the username
  • In a URL parameter: ?id=1' — check for error messages
  • Try: ?id=1 OR 1=1 — does it return more results?
  • Look for database error messages in the response
  • Use sqlmap for automated testing
👁️ What to Look For
  • Database error messages (MySQL, PostgreSQL, SQL Server errors)
  • Different content returned with injected queries
  • Authentication bypass (logging in as admin without a password)
  • Slower response times with time-based payloads
💥 Impact: Complete database compromise: read/modify/delete all data, bypass authentication, potentially execute system commands. SQL injection is one of the most dangerous web vulnerabilities.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free) 🔧 sqlmap (free)
📝 Report Tip: Show the injection payload, the database error or altered response, and what data could be extracted. NEVER include actual database dumps in your report.
15

📤 File Upload Vulnerabilities

Upload malicious files that the server processes
★★☆ 💰💰 ▼
📖 What Is It?

When a website lets you upload files (avatars, documents, images), it should validate the file type and content. If it doesn't, you can upload a malicious file — like a web shell that gives you control of the server. It's like a mailroom that accepts "fragile" packages without checking what's inside.

🔍 When to Look
  • Profile picture uploads
  • Document upload features
  • Resume/CV upload forms
  • Any form with a file input
🧪 How to Test
  • Try uploading a .php, .jsp, or .aspx file directly
  • If blocked, rename to .jpg.php or .php.jpg
  • Try changing the Content-Type header to image/jpeg
  • Add a GIF header (GIF89a) to a PHP file
  • Check if uploaded files are accessible via URL
  • Upload an HTML file with <script>alert(1)</script>
👁️ What to Look For
  • Malicious files uploaded successfully
  • Uploaded files are accessible and executed
  • No file type validation on the server
💥 Impact: Remote code execution, full server compromise, data theft, defacement. One of the most critical vulnerabilities when exploitable.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free) 📝 Text Editor
📝 Report Tip: Show the uploaded file, the URL to access it, and demonstrate code execution (like displaying server info). Never upload actual malware.
16

🧠 Business Logic Flaws

Abuse the application's logic to get unintended results
★★☆ 💰💰 ▼
📖 What Is It?

Business logic flaws happen when the application's rules can be abused. Like using a coupon 100 times, buying an item for negative quantity to get a refund, or exploiting a race condition to get free items. No code injection needed — just thinking creatively about how the app works.

🔍 When to Look
  • Shopping carts (quantity, price, discount codes)
  • Registration flows (referral bonuses, free trials)
  • Payment processing (coupons, credits, wallets)
  • Game mechanics (points, rewards, levels)
  • Any workflow with business rules
🧪 How to Test
  • Try negative quantities in shopping carts
  • Use the same coupon code multiple times
  • Modify prices in the request (client-side pricing)
  • Try applying multiple discounts
  • Send the same request multiple times quickly (race condition)
  • Change the order of steps in a workflow
👁️ What to Look For
  • Negative prices or quantities accepted
  • Discounts can be stacked or reused
  • Free items obtained through logic abuse
  • Race conditions give duplicate credits
💥 Impact: Financial loss, free products/services, account abuse, and fraud. These are often the highest-paying bugs because they directly affect revenue.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Clearly explain the business logic, show step-by-step how to reproduce, and quantify the financial impact (e.g., "obtained $500 in credits for free").
17

⏱️ Rate Limiting Bypass

Send unlimited requests when there should be a limit
★★☆ 💰 ▼
📖 What Is It?

Rate limiting stops you from sending too many requests too fast. Bypassing it means you can brute-force passwords, spam forms, or abuse features without restriction. It's like a bouncer who counts entries per person — but you can trick them by wearing different hats.

🔍 When to Look
  • Login pages with rate limiting (after 5 failed attempts)
  • OTP/code verification endpoints
  • Password reset forms
  • API endpoints with request limits
🧪 How to Test
  • Send requests until rate limited (usually 429 error)
  • Try changing the IP address header: X-Forwarded-For
  • Try changing the request body slightly
  • Use different User-Agent strings
  • Try sending requests in parallel (race condition)
  • Change the request order or parameters
👁️ What to Look For
  • Rate limit resets after changing headers
  • Can send unlimited requests with modified parameters
  • No rate limiting at all on sensitive endpoints
💥 Impact: Enables brute-force attacks on passwords, OTP bypass, account enumeration, and denial of service. Critical for authentication endpoints.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 curl
📝 Report Tip: Show the bypass technique, demonstrate sending requests past the rate limit, and explain what could be brute-forced.
18

🔐 2FA Bypass

Skip or bypass two-factor authentication
★★☆ 💰💰 ▼
📖 What Is It?

Two-factor authentication (2FA) asks for a code from your phone after your password. Bypassing it means getting in without that code. Common bypasses include skipping the 2FA step, brute-forcing the 2FA code, or bypassing through the API directly.

🔍 When to Look
  • Any site with 2FA enabled
  • Login flows, especially after password entry
  • Password reset flows that skip 2FA
  • API endpoints that don't enforce 2FA
🧪 How to Test
  • Login with password, then try to access the dashboard directly (skip the 2FA page)
  • Try the API endpoint directly with valid credentials
  • Brute-force the 2FA code (usually 4-6 digits = 10,000-1,000,000 combinations)
  • Try the password reset flow — does it skip 2FA?
  • Check if 2FA is enforced on all endpoints
👁️ What to Look For
  • Can access account without entering 2FA code
  • 2FA code is accepted on multiple attempts (no lockout)
  • API responses don't require 2FA verification
💥 Impact: Complete account takeover even when 2FA is enabled. This defeats one of the most important security controls.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Explain the bypass method, show you can access the account without the 2FA code, and emphasize the account takeover impact.

🔑 Password Reset Vulnerabilities

Abuse password reset flow to take over accounts
★★☆ 💰💰 ▼
📖 What Is It?

Password reset flows have many steps — and each step can have bugs. Common issues: the reset token is predictable, the token doesn't expire, the email address can be changed in the request, or the token is exposed in the URL. It's like a lock that can be picked because the key is left under the mat.

🔍 When to Look
  • Any "Forgot Password" feature
  • Reset token in URL (can be shared or leaked)
  • Can change the email in the request
  • Token doesn't expire or can be reused
  • Reset code sent to phone can be brute-forced
🧪 How to Test
  • Request a password reset and check the email
  • Look at the reset URL — is the token guessable?
  • Try using the token after setting a new password (reuse)
  • Intercept the reset request and change the email to your own
  • Try a new password reset — does the old token still work?
  • Check if reset tokens are sequential or predictable
👁️ What to Look For
  • Reset token in URL (visible, shareable, leaked via Referer header)
  • Token reuse after password change
  • Email can be changed in the request
  • Weak token generation (sequential, timestamp-based)
💥 Impact: Account takeover by resetting any user's password. Critical security flaw that can affect all users.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Show the full reset flow, explain the flaw, demonstrate taking over an account, and recommend using cryptographically random tokens that expire.

🏠 Host Header Injection

Manipulate the Host header to poison responses
★★☆ 💰💰 ▼
📖 What Is It?

The Host header tells the server which domain you're visiting. If the server trusts this header without validation, you can manipulate it to poison password reset emails, bypass access controls, or cache poison responses. It's like telling the post office a different return address on your letter.

🔍 When to Look
  • Password reset links (does the Host header affect the reset URL?)
  • Pages that reflect the Host header in content
  • Proxy environments
  • Virtual host configurations
🧪 How to Test
  • Intercept a request and change the Host header to evil.com
  • Request a password reset — does the email link use the malicious host?
  • Check if the response content changes with the Host header
  • Try with different Host values: evil.com, evil.com:80, evil.com\@good.com
👁️ What to Look For
  • Password reset links contain the malicious host
  • Response content reflects the Host header
  • Cached responses serve poisoned content
💥 Impact: Password reset poisoning (steal reset tokens), web cache poisoning, XSS via poisoned responses, and authentication bypass.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 curl
📝 Report Tip: Show the modified Host header, the poisoned response/email, and explain the attack scenario.

💾 Stored XSS

Permanent script injection that affects all visitors
★★★ 💰💰💰 ▼
📖 What Is It?

Stored XSS is when your injected script is saved in the database and runs for EVERYONE who views that page. Unlike reflected XSS (which needs a crafted link), this one is persistent — like graffiti that everyone who visits the page can see. It's the most dangerous type of XSS.

🔍 When to Look
  • Comment sections, forum posts, reviews
  • Profile fields (bio, username, description)
  • Chat/messaging features
  • Any input that is saved and displayed to other users
🧪 How to Test
  • Enter <script>alert(1)</script> in a comment/profile field
  • Save it and reload the page — does the alert pop up?
  • If blocked, try: <img src=x onerror=alert(1)>
  • Check other user profiles — can you inject into their fields?
  • Test all input fields that are displayed to other users
👁️ What to Look For
  • Script executes when ANY user views the page
  • No input sanitization on stored data
  • Admin users also affected
💥 Impact: Steal session cookies of all users viewing the page, deface the website, spread worms, steal sensitive data, and perform actions as any user. Critical severity.
🛠️ Tools Needed
🌐 Browser 🔧 Burp Suite (free)
📝 Report Tip: Show the injection point, demonstrate the script executing for other users, and explain the impact on all users. This is a critical finding — be thorough.

🎯 Blind SQL Injection

SQL injection with no visible error output
★★★ 💰💰💰 ▼
📖 What Is It?

Blind SQL injection is when the database is vulnerable but doesn't show errors. You have to infer data bit-by-bit using true/false questions or time delays. It's like playing "20 Questions" with a database — you ask yes/no questions to guess the data one character at a time.

🔍 When to Look
  • Same places as regular SQL injection
  • When you get no error messages after injecting
  • When the page changes slightly (true vs false conditions)
  • When response times differ based on your input
🧪 How to Test
  • Boolean-based: ?id=1 AND 1=1 vs ?id=1 AND 1=2
  • Check if responses differ between true and false conditions
  • Time-based: ?id=1 AND SLEEP(5) — does the response take 5 seconds?
  • Use sqlmap: sqlmap -u "http://target/?id=1" --batch
👁️ What to Look For
  • Page content differs between true/false conditions
  • Response time varies with SLEEP/WAITFOR payloads
  • Boolean-based differences in page content
💥 Impact: Same as SQL injection but harder to detect. Complete database compromise, data extraction, authentication bypass. Often rated higher because it's harder to find.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 sqlmap (free)
📝 Report Tip: Show the boolean/time difference, demonstrate data extraction, and use sqlmap output as proof.

☁️ SSRF to Cloud Metadata

Access cloud credentials through SSRF
★★★ 💰💰💰 ▼
📖 What Is It?

Cloud services (AWS, GCP, Azure) have a special internal URL that provides credentials and configuration. If you have SSRF and can reach this URL, you get the cloud credentials — which can be used to access the entire cloud infrastructure. It's like finding the master key to the entire building hidden in a service closet.

🔍 When to Look
  • When you find a basic SSRF vulnerability
  • Targets running on AWS, GCP, or Azure
  • The SSRF can reach internal IP addresses
🧪 How to Test
  • AWS: http://169.254.169.254/latest/meta-data/
  • GCP: http://169.254.169.254/computeMetadata/v1/
  • Azure: http://169.254.169.254/metadata/instance?api-version=2021-02-01
  • Try accessing these through the SSRF vector
  • If metadata is returned → CRITICAL finding!
👁️ What to Look For
  • Cloud metadata returned (IAM credentials, instance info)
  • Access to S3 buckets, databases, or other cloud services
  • Full cloud infrastructure compromise possible
💥 Impact: CRITICAL. Full cloud infrastructure compromise: access all databases, storage, compute resources. Can lead to data breaches affecting millions of users. Often the highest-paying bug class.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 curl

💀 RCE (Remote Code Execution)

Execute arbitrary commands on the server
★★★ 💰💰💰 ▼
📖 What Is It?

RCE means you can run any command on the server. This is the most critical vulnerability possible — it means the attacker has complete control. It's like having the admin password to every computer in the building. This is the "holy grail" of bug bounties.

🔍 When to Look
  • Command injection in user inputs
  • Deserialization vulnerabilities
  • Template injection (SSTI)
  • File upload to webshell
  • SSRF that can reach internal services
  • Known CVEs in used software
🧪 How to Test
  • Try command injection: ; whoami, | whoami, $(whoami)
  • Test for template injection: {{7*7}} — if 49 appears, try {{config}}
  • Check for deserialization: look for serialized objects in requests
  • Use known exploits for software versions found in information disclosure
👁️ What to Look For
  • Command output appears in the response
  • Server responds to injected commands
  • Arbitrary file read/write
💥 Impact: CRITICAL. Complete server compromise. The attacker can read all data, install backdoors, pivot to other systems, and cause maximum damage. This is typically the highest-paying vulnerability.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 curl 🌐 Browser
📝 Report Tip: Show the command executed and output. NEVER cause damage, modify data, or access other users' data. Be minimal in your proof. Report immediately.

🔓 Authentication Bypass

Access accounts without valid credentials
★★★ 💰💰💰 ▼
📖 What Is It?

Authentication bypass means getting past the login screen without knowing the password. This could be through SQL injection, logic flaws, JWT manipulation, or broken session management. It's like finding a key that opens every door in a building.

🔍 When to Look
  • Login pages with unusual behavior
  • Session management (cookies, tokens)
  • Remember-me functionality
  • Passwordless login (magic links, OTP)
  • OAuth/SSO implementations
🧪 How to Test
  • Try SQL injection in login: admin' OR '1'='1
  • Check session tokens for predictability
  • Try accessing authenticated pages without logging in
  • Modify session cookies
  • Check if "remember me" tokens are guessable
👁️ What to Look For
  • Access to authenticated areas without credentials
  • Session tokens that are predictable or weak
  • Authentication checks that can be bypassed
💥 Impact: CRITICAL. Full account takeover, access to all user data, potential for mass exploitation. This is a critical security failure.

🔧 Burp Suite (free) 🌐 Browser

🎫 JWT Attacks

Manipulate JSON Web Tokens to impersonate users
★★★ 💰💰💰 ▼
📖 What Is It?

JWTs (JSON Web Tokens) are used for authentication. They have three parts: header, payload, and signature. Common attacks include: changing the algorithm to "none" (bypassing signature), using the server's public key to sign, or modifying the payload to change your user ID. It's like forging an ID card by changing the photo.

🔍 When to Look
  • Applications that use JWT for authentication
  • Token-based auth (look for Bearer tokens)
  • APIs with JWT in headers
🧪 How to Test
  • Decode the JWT (base64 decode the first two parts)
  • Change the algorithm to "none" and remove the signature
  • Change the "user" or "sub" claim to another user
  • Try the RS256→HS256 algorithm confusion attack
  • Check if the token is expired or has weak validation
👁️ What to Look For
  • Modified tokens are accepted
  • Algorithm change bypasses validation
  • Payload modification grants unauthorized access
💥 Impact: Account impersonation, privilege escalation, access to any user's account. Critical for JWT-based authentication systems.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 jwt_tool (free)
📝 Report Tip: Show the modified JWT, demonstrate accessing another user's account, and explain the authentication bypass.

🔌 API Security (BOLA & Mass Assignment)

Exploit broken API access controls and object binding
★★★ 💰💰💰 ▼
📖 What Is It?

BOLA (Broken Object Level Authorization) is like IDOR but for APIs — you can access any user's data through API endpoints. Mass Assignment is when you can set fields you shouldn't be able to (like making yourself an admin by adding "admin": true to the request). Both are top API vulnerabilities.

🔍 When to Look
  • REST/GraphQL APIs (check the network tab for XHR requests)
  • Endpoints with user IDs: /api/users/123
  • Registration or profile update endpoints
  • Any endpoint that accepts JSON
🧪 How to Test
  • For BOLA: Change the user ID in API requests to access other users' data
  • For Mass Assignment: Add extra fields in registration/update requests (e.g., "role": "admin", "isVerified": true)
  • Check if the API has proper authorization checks
  • Use a tool like Postman or curl to test API endpoints
👁️ What to Look For
  • Can access other users' data through API
  • Can modify privileged fields (role, permissions, verified status)
  • No rate limiting on API endpoints
💥 Impact: Data breach, privilege escalation, account takeover. APIs are the #1 attack vector in modern applications.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 Postman (free) 🔧 curl
📝 Report Tip: For BOLA: show the API request with changed IDs. For Mass Assignment: show the extra fields and their effect on the account.

🔮 GraphQL Vulnerabilities

Exploit GraphQL introspection and query features
★★★ 💰💰 ▼
📖 What Is It?

GraphQL is an API query language that lets clients ask for exactly what they need. If not secured properly, attackers can use "introspection" to discover the entire API schema, find hidden fields, bypass authorization, and run expensive queries (denial of service).

🔍 When to Look
  • Any GraphQL endpoint (usually /graphql or /api/graphql)
  • Try sending a GET request with ?query={__schema{types{name}}}
  • Look for GraphQL in the network tab of DevTools
🧪 How to Test
  • Enable introspection: { __schema { types { name } } }
  • Look for hidden queries and mutations
  • Try querying other users' data
  • Test for IDOR through GraphQL queries
  • Send deeply nested queries for DoS
  • Use tools: InQL, GraphQL Raider (Burp extension)
👁️ What to Look For
  • Introspection enabled (full schema disclosure)
  • Can access other users' data through queries
  • Hidden admin queries or mutations
💥 Impact: Data exposure, authorization bypass, denial of service, and full API compromise depending on the vulnerabilities found.
🛠️ Tools Needed
🔧 Burp Suite (free) 🔧 InQL (free) 🔧 GraphQL Raider
📝 Report Tip: Show the introspection result, demonstrate accessing unauthorized data, and list the exposed API schema.

🏝️ Subdomain Takeover

Claim an abandoned subdomain pointing to a service you control
★★★ 💰💰💰 ▼
📖 What Is It?

When a company stops using a service (like a Heroku app or GitHub Pages) but forgets to remove the DNS record pointing to it, anyone can claim that subdomain. The DNS still says "blog.company.com → herokuapp.com" but nobody owns that Heroku app anymore. You create it and now you control blog.company.com.

🔍 When to Look
  • Any subdomain of the target
  • DNS records pointing to external services (Heroku, AWS, GitHub, Shopify)
  • Services that show "No such app" or "Page not found" messages
🧪 How to Test
  • Enumerate subdomains: use subfinder, amass, or crt.sh
  • Check each subdomain's DNS records
  • Look for CNAME records pointing to external services
  • Visit each subdomain and check for "unclaimed" messages
  • Use subjack or can-i-take-over-xyz to automate detection
  • If the service is claimable → Subdomain Takeover!
👁️ What to Look For
  • CNAME pointing to a service you can register
  • "No such app" or "Not found" on external services
  • GitHub Pages, Heroku, AWS S3 with unclaimed domains
💥 Impact: Phishing (using the legitimate subdomain), cookie theft, session hijacking, XSS on a trusted domain, and full account takeover.
🛠️ Tools Needed
🔧 subfinder (free) 🔧 subjack (free) 🌐 crt.sh (free)
📝 Report Tip: Show the DNS records, demonstrate the service is claimable (DO NOT claim it), and explain the phishing/cookie theft impact.

🚚 HTTP Request Smuggling

Confuse servers by sending ambiguous HTTP requests
★★★ 💰💰💰 ▼
📖 What Is It?

HTTP Request Smuggling exploits differences in how front-end (CDN/proxy) and back-end servers interpret HTTP requests. One server thinks one request ends here, but the other server thinks it ends there — allowing you to "smuggle" a second request inside the first. It's like hiding a second letter inside an envelope that the mailroom doesn't open.

🔍 When to Look
  • Targets behind a CDN or load balancer (Cloudflare, AWS ALB, etc.)
  • HTTP/1.1 and HTTP/2 coexistence
  • Any target with front-end/back-end architecture
🧪 How to Test
  • CL.TE: Send a request with both Content-Length and Transfer-Encoding headers
  • TE.CL: Swap which server trusts which header
  • TE.TE: Use obfuscated Transfer-Encoding headers
  • Use Burp Suite's HTTP Request Smuggler extension
  • Watch for differential responses (200 vs 404, etc.)
👁️ What to Look For
  • Different responses from front-end vs back-end
  • Can poison other users' requests
  • Can bypass security controls
💥 Impact: Request smuggling, cache poisoning, session hijacking, XSS, and credential theft. Can affect all users of the service.
🛠️ Tools Needed
🔧 Burp Suite Pro 🔧 HTTP Request Smuggler 🔧 curl
📝 Report Tip: Show the smuggled request, demonstrate the impact (hijacking another user's request), and recommend consistent parsing across servers.

☁️ Cloud Vulnerabilities

Most websites run on cloud servers (AWS, GCP, Azure). When companies misconfigure cloud settings, attackers can steal credentials, access databases, and take over entire infrastructures. Cloud bugs pay $2,000–$20,000+.

🪣

Open S3 Buckets

Exposed cloud storage containing sensitive data
Advanced ▼

🧠 What is it?

Amazon S3 is a cloud storage service — like a hard drive on the internet. Companies store backups, customer data, source code, and API keys there. When bucket permissions are set wrong, anyone on the internet can read (or write to) the files. It's one of the most common and impactful cloud misconfigurations.

📝 Analogy

A company puts sensitive files in a cloud storage box but sets the permission to "anyone with the link can view." An attacker finds the link and downloads everything — customer records, passwords, source code.

🎯 Where to look

  • JavaScript files — open DevTools (F12), search source for "s3.amazonaws.com" or "s3://"
  • Google dork: site:s3.amazonaws.com "company-name"
  • GitHub repos — search for S3 URLs in public code
  • Mobile apps — intercept API calls, look for S3 URLs
  • Common bucket names: company-backups, company-assets, company-staging, company-dev

🔬 How to test

1
Find an S3 URL in JavaScript source or Google dorks: https://BUCKETNAME.s3.amazonaws.com
2
Visit the URL — if you see an XML file listing files, the bucket is open!
3
Try downloading files: aws s3 ls s3://BUCKET-NAME --no-sign-request
4
If you can list and download files → report it. If you can UPLOAD files → Critical severity!

💉 What to try

-- Common bucket patterns
https://company-name.s3.amazonaws.com
https://company-backups.s3.amazonaws.com
https://company-staging.s3.amazonaws.com

-- List bucket contents (if open)
aws s3 ls s3://BUCKET-NAME --no-sign-request

-- Download files
aws s3 cp s3://BUCKET-NAME/secret-file.txt . --no-sign-request

-- Test write access (CRITICAL if it works!)
echo "test" > test.txt
aws s3 cp test.txt s3://BUCKET-NAME/test.txt --no-sign-request
📝 Report Tip: List the sensitive files you found. Mention the data type (customer PII, source code, credentials). If write access is possible, escalate severity — this could allow website defacement or malware injection.
📡

Cloud Metadata Exploitation

Steal server credentials via the metadata service
Advanced ▼

🧠 What is it?

Cloud servers have a special internal URL (169.254.169.254) that returns info about the server — including temporary security credentials (access keys, tokens). If you find an SSRF vulnerability, you can make the server request this URL and steal its cloud credentials. Those credentials can access everything the server has access to.

📝 Analogy

Every cloud server has a "server ID card" on its desk. The ID card gives access to the cloud account. Trick the server into showing you its ID card (via SSRF) and you get its access keys — letting you impersonate the server.

🎯 Where to look

  • Any SSRF vulnerability — this is the escalation path for SSRF
  • "Import from URL" features, image loaders, PDF generators
  • Targets on AWS/GCP/Azure — check with wappalyzer or DNS records
  • CNAME records pointing to cloud providers: dig target.com

🔬 How to test

1
Find an SSRF vulnerability first (see SSRF section above)
2
Set up a listener: go to webhook.site (free)
3
Try accessing the metadata URL via SSRF: http://169.254.169.254/latest/meta-data/
4
If you get a response, explore: /iam/security-credentials/ to find role names
5
The role's temporary credentials give you cloud access — report immediately!

💉 Metadata URLs

-- AWS Metadata (IMDSv1)
http://169.254.169.254/latest/meta-data/
http://169.254.169.254/latest/meta-data/iam/security-credentials/
http://169.254.169.254/latest/user-data/

-- GCP Metadata
http://metadata.google.internal/computeMetadata/v1/
http://169.254.169.254/computeMetadata/v1/instance/service-accounts/default/token

-- Azure Metadata
http://169.254.169.254/metadata/instance?api-version=2021-02-01

-- Combine with SSRF:
https://target.com/fetch?url=http://169.254.169.254/latest/meta-data/
📝 Report Tip: Show that you reached the metadata endpoint. DO NOT use the stolen credentials. Explain the blast radius — what services could be accessed. This is typically Critical severity.

Exposed API Keys and Credentials

Leaked cloud keys in source code and config files
Intermediate ▼

🧠 What is it?

Developers sometimes hardcode API keys, database passwords, and cloud credentials into source code, JavaScript files, or config files. If these end up in public repos or client-side code, attackers can use them to access cloud services, databases, and internal systems.

🎯 Where to look

  • JavaScript source code — view page source, search for keys
  • Public GitHub repos — search for key prefixes
  • Environment files — /.env, /config.js, /settings.json
  • Mobile app binaries — decompile and search for strings
  • Paste sites — developers sometimes paste keys by accident

💉 Key prefixes to search for

-- AWS Keys (search GitHub for "AKIA")
AKIA[0-9A-Z]{16}

-- GCP Keys
AIza[0-9A-Za-z_-]{35}

DefaultEndpointsProtocol=https;AccountName=...

-- Private Keys
-----BEGIN RSA PRIVATE KEY-----
-----BEGIN OPENSSH PRIVATE KEY-----

-- GitHub PAT
ghp_[A-Za-z0-9]{36}

-- Slack token
xox[baprs]-[A-Za-z0-9-]+

🧠 What is it?

Advanced ▼

🧠 What is it?

Many modern apps use serverless functions (AWS Lambda, Azure Functions) instead of full servers. These functions store secrets in environment variables, have overprivileged IAM roles, or are accessible via public URLs without authentication.

-- Test function URLs for:

-- Path traversal

🎯 Your First Bug Bounty Program

Ready to start earning? Follow these steps for your first program.

Sign up at HackerOne (hackerone.com) or Bugcrowd (bugcrowd.com). Both are free and trusted by thousands of companies. Start with public programs that welcome beginners.

Every program has a "scope" (which domains/properties you can test) and "rules" (what's allowed). READ THEM CAREFULLY. Testing out of scope can get you banned or in legal trouble.

Map out the target: find subdomains, endpoints, features, and technologies. Use crt.sh for subdomains, Wappalyzer for technology detection, and your browser for manual exploration.

Don't try everything at once. Pick ONE vulnerability class (like IDOR or XSS) and test every feature with it. You'll learn faster and find more bugs.

Take screenshots, save requests in Burp, and write detailed steps to reproduce. Good documentation = faster triage = faster payout.

Write a clear, professional report. Explain the vulnerability, show the impact, and provide reproduction steps. Be respectful and patient.

📝 How to Write a Report

A good report gets you paid faster. Use this template.

📋 Bug Report Template

[Vulnerability Type] in [Feature/Endpoint] — e.g., "IDOR in /api/user/profile allows accessing other users' data"

Rate the severity yourself: Critical / High / Medium / Low / Info. This shows you understand the impact.

One paragraph explaining what the vulnerability is and why it matters. Keep it simple — the triager might not be technical.

Numbered steps that anyone can follow: "1. Create account A and B. 2. Log in as A. 3. Go to /user/1002. 4. See account B's data."

Explain what an attacker could do: "An attacker can access any user's private data including emails, addresses, and payment information."

Include screenshots, videos, or proof-of-concept code. More evidence = faster triage.

Suggest how to fix it: "Implement proper authorization checks on the /user endpoint."

⚠️ Common Mistakes to Avoid

Learn from others' mistakes. Avoid these pitfalls.

🚫

Testing Out of Scope

Always read the program scope. Testing unauthorized domains or features can get you banned and even in legal trouble.

🚫

Not Taking Screenshots

Always capture evidence before it disappears. A picture is worth a thousand words — and a faster payout.

🚫

Submitting Duplicate Reports

Check if the bug has already been reported. Search the program's resolved reports before submitting.

🚫

Being Aggressive or Demanding

Stay professional. Triagers are people too. Be patient, be respectful, and follow up politely if needed.

🚫

Ignoring Program Rules

Each program has specific rules. Some don't allow social engineering, some have rate limits. Violating them = disqualification.

🚫

Trying to Find Everything at Once

Focus on one bug type. Master it, find bugs with it, then move to the next. Spreading thin leads to finding nothing.

🚫

Not Practicing First

Use PortSwigger Labs, DVWA, or HackTheBox to practice before testing real programs. Practice builds skill and confidence.

🚫

Reporting Low-Severity Issues Without Context

Missing headers or information disclosure alone are low impact. Explain HOW they can be combined with other bugs for higher impact.

🎯 Where to Find Programs

These platforms host bug bounty programs from top companies.

🏴 HackerOne

The largest bug bounty platform. Thousands of programs from startups to Fortune 500 companies. Great for beginners with public programs.

🏴 Bugcrowd

Another major platform with diverse programs. Has a "Priority Brief" system for new researchers. Good starting point.

🔒 Intigriti

European-based platform with quality programs. Known for monthly challenges and good researcher support.

🏢 Synack

Private, invite-only platform for experienced researchers. Higher pay but harder to get into.

🔐 YesWeHack

European platform with strong government and enterprise programs. Growing rapidly.

💡 Open Bug Bounty

Non-commercial, transparent platform. Good for beginners. Focuses on coordinated disclosure.

🏢 Google VRP

Google's own bug bounty program. Covers all Google products, Android, and Chrome. Well-paying and well-organized.

🍎 Apple Security Bounty

Apple's program for iOS, macOS, and web vulnerabilities. High rewards but strict rules.
